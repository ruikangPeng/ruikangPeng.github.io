白盒测试的测试用例设计方法

一、什么是白盒测试

白盒测试也称结构测试或逻辑驱动测试，是把测试对象看作一个透明白色的盒子，测试人员依据程序内部逻辑结构相关信息，设计或选择测试用例，对程序所有逻辑路径进行测试。

所以白盒测试是一种基于代码的测试，测试者必须检查程序的内部结构，从检查程序的逻辑着手，得出测试数据。

一般来说，白盒测试是开发做单元测试用，但是也有是测试自己做的，那么对测试的能力要求比较高。

二、测试用例设计方法

白盒的测试用例设计方法有六种，按照强度由低到高分别为：语句覆盖、分支覆盖（判定覆盖）、条件覆盖、判定条件覆盖、条件组合覆盖、路径覆盖。

以下面这段程序举例说明：

1、语句覆盖

要求保证程序中每个语句至少被执行一次。

满足语句覆盖设计一条测试用例即可：A = true，B = true，C = true，D = 随意

这个方法仅仅考虑了对代码中的执行语句进行覆盖而没有考虑各种条件和分支。

2、分支覆盖（判定覆盖）

要求程序中每个判断取真和取假至少经历一次。

具体说来，满足分支覆盖要考虑：

  1. A and B == true
  2. A and B == false
  3. C or D == true
  4. C or D == false

由此，我们设计两条测试用例即可：

  1. A = true，B = true，C = true，D = 随意（这条用例就满足了上述1、3两条考虑）
  2. A = false，B = 随意，C = false，D = false（这条用例就满足了上述2、4两条考虑）

3、条件覆盖

要求能使每个判定中的每个条件都获得可能的取值。即每个条件至少有一次真值和假值。

具体说来，满足条件覆盖要考虑八种情况：

  1. A = true / false
  2. B = true / false
  3. C = true / false
  4. D = true / false

由此，我们设计两条测试用例即可：

  1. A = true，B = true，C = true，D = true
  2. A = false，B = false，C =false，D = false

4、判定条件覆盖

要求可以使得程序判断中每个条件所有的可能取值至少执行一次，同时每个判断本身所有的结果也要至少执行一次。

具体说来，满足判定条件覆盖要同时考虑分支覆盖和条件覆盖考虑的内容。

由此，我们设计两条测试用例就可以满足覆盖需求：

  1. A = true，B = true，C =false，D = false
  2. A = false，B = false，C = true，D = true

5、条件组合覆盖

要求可以使得程序每个判定中的各个条件的各种可能组合都至少出现一次。

具体说来，满足条件组合覆盖要考虑A、B、C、D的所有取值。也要考虑A和B的所有组合、C和D的所有组合。

由此，我们需要设计四条测试用例才能满足覆盖需求：

  1. A = true，B = true，C = true，D = true
  2. A = true，B = false，C =true，D = false
    1. A = false，B = true，C = false，D = true
    1. A = false，B = false，C = false，D = false

6、路径覆盖

要求覆盖程序中所有可能的路径。即穷举所有，这个实际操作性不强。

三、对白盒测试的评价

  1. 优点
  • 迫使测试人员去仔细思考软件的实现
  • 帮助软件测试人员增大代码的覆盖率。提高代码的质量，发现代码中隐藏的问题
  1. 缺点
  • 昂贵: 系统庞大时，测试开销会非常大;测试人员要求比较高，人才昂贵。
  • 测试基于代码，只能测试开发人员做的对不对，而不能知道设计是否正确，可能会漏掉一些功能需求

四、总结

以上简单描述了几种不用的白盒测试方法的原则和优劣。在实际的操作中，要正确使用白盒测试的代码覆盖方法，就要从代码分析和代码调研入手，根据调研的结果，可以选择上述方法中的某一种，或者好几种方法的结合，设计出高效的测试用例，尽可能全面地覆盖到代码中的每一个逻辑路径，只有适当的测试方法，才能帮助我们从代码的角度对产品做到最好的测试。
